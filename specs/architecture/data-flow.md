# データフロー

## 概要

情報収集から投稿反応分析まで、データがシステム内をどう流れるかを記述する。

## フロー全体図

```
[外部情報源]
    │
    │ (1) 情報取得
    ▼
┌─────────────────┐
│ info-collector  │
└────────┬────────┘
         │
         │ (2) 正規化・保存
         ▼
┌─────────────────┐
│ raw_materials   │
└────────┬────────┘
         │
         │ (3) 素材読み取り
         ▼
┌─────────────────┐
│ post-generator  │
└────────┬────────┘
         │
         │ (4) ドラフト生成・保存
         ▼
┌─────────────────┐
│ drafts          │
└────────┬────────┘
         │
         │ (5) レビュー対象取得
         ▼
┌─────────────────┐
│ post-operator   │← (6) 人間レビュー・投稿
└────────┬────────┘
         │
         │ (7) 投稿記録
         ▼
┌─────────────────┐
│ posts           │
└────────┬────────┘
         │
         │ (8) メトリクス紐付け
         ▼
┌─────────────────┐
│ post_metrics    │← (9) 反応データ入力 (dashboard経由)
└────────┬────────┘
         │
         │ (10) 集計
         ▼
┌─────────────────┐
│ dashboard       │
└─────────────────┘
```

## 各フローの説明

### (1) 情報取得
info-collector が外部情報源を取得する。

### (2) 正規化・保存
情報源ごとに異なるレスポンス形式を共通スキーマに正規化し、raw_materials に保存する。重複検出して重複素材はスキップする。

### (3) 素材読み取り
post-generator が未処理の raw_materials を取得する。

### (4) ドラフト生成・保存
各素材に対して5ステップパイプラインを実行し、drafts に保存する。1素材から複数ドラフト（異なる型）を生成することがある。

### (5) レビュー対象取得
post-operator がレビュー待ちの drafts を提示する。

### (6) 人間レビュー・投稿
オペレータが drafts を確認する。承認 → 人間がXに手動投稿、却下 → 却下ステータスに。

### (7) 投稿記録
投稿後、オペレータが投稿情報を post-operator に入力し、posts に保存する。

### (8) メトリクス紐付け
posts と post_metrics は1対多。時系列でメトリクスを複数記録できる。

### (9) 反応データ入力
- 初期: dashboard のUI経由で手動入力
- 将来: 自動取得（切替タイミングは実装フェーズで判断）

### (10) 集計
dashboard が posts と post_metrics を結合して型別・期間別で集計する。

## エラー時のデータ扱い

### info-collector 失敗時
リトライはせず、次回の定期実行に任せる。エラーログを記録する。

### post-generator 失敗時
該当 raw_material をエラーステータスに更新し、手動でリトライ可能とする。

### post-operator の投稿URL入力ミス
訂正機能を提供する。
