# 判断を保留する実装論点

実装開始時にRyugoさんとの相談で決定する項目の集約。

## アーキテクチャ・インフラ

- Docker構成の具体設計（compose構成、ボリューム、ネットワーク）
- ホスティング方式
- dashboard のバイナリ化戦略
- 各プロセスの起動・スケジュール方式
- 具体的なフレームワーク・ライブラリ選定
- フロントエンドのUIライブラリ・スタイリング手法選定
- 投稿ドラフト生成LLMの選定

## データ

- raw_materials の重複検出ロジックの厳密性
- 正規化ルールの詳細
- 各データの自動削除期間
- post_metrics のスナップショット取得タイミング
- 具体的なカラム型、制約、インデックス設計
- マイグレーション戦略
- JSONカラムの具体スキーマ
- accounts / info_sources テーブルの詳細設計
- 論理削除の具体実装

## コンポーネント

### info-collector
- API制限ヒット時の待機・再試行方針
- 情報源ごとの取得パラメータの具体値

### post-generator
- 各ステップのLLMプロンプトの具体内容
- Step3 選別基準の具体的な閾値
- Step4 型選択ロジック（ルールベース / LLM判定 / ハイブリッド）
- 生成失敗時のリトライ戦略
- メディア候補の取得方法

### post-operator
- CLIのUI/UX設計
- ドラフト編集機能の有無と範囲
- 承認済みドラフトの保持期間

### dashboard
- 認証方式（ローカル限定 / Basic認証 / なし）
- APIエンドポイントのURL設計、クエリパラメータ、ページネーション方針
- レスポンスの具体JSONスキーマ
- エラーコード体系
- リアルタイム通信要否
- UIコンポーネント構成、レスポンシブ対応、テーマ、ショートカット
- データ更新頻度

## 運用

- 「話題性の指標」の具体的な判定基準
- Step3 選別の閾値
- ペルソナ設定の詳細
- Grok API のクエリ具体文言
- 楽天ブックスAPIの検索条件
- 情報源追加・削除の判断基準
- 事前評価チェックリストの項目定義
- メトリクス入力のタイミング
- スケジュール投稿ツールの採用可否
- 投稿予定時刻の post-operator 側での上書き可否
- 休日・祝日のスケジュール調整方針

## フェーズ

- 各フェーズの所要期間
- 並行着手の可否
- 失敗時の打ち切り・再設計の判断基準
- X API 切り替えタイミングとその判断基準
- 反応データ取得方法の確定（手動 / CSV / API）
- 「型」の改善ループの具体フロー
